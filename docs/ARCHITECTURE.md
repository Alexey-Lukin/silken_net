## Data Flow (Gaia 2.0 Pipeline)

1. **Sensor Acquisition** вҖ” Soldier reads impedance (EIS), temperature, acoustic events (piezo), and supercapacitor voltage.
2. **TinyML Audio Classification** вҖ” On-device neural network classifies audio: silence / wind / cavitation / chainsaw.
3. **mruby Bio-Contract** вҖ” Lorenz attractor computation on-device produces `growth_points` and `bio_status`.
4. **Bit-Packing & Encryption** вҖ” 16-byte payload packed, AES-256-ECB encrypted, transmitted via LoRa.
5. **Queen Reception** вҖ” Queen decrypts, caches via CIFO (Closest In, Farthest Out), batches up to 50 trees.
6. **CoAP Uplink** вҖ” Queen sends binary batch via CoAP PUT over LTE/Starlink to Rails backend (Akash or GCP).
7. **Backend Unpacking** вҖ” `UnpackTelemetryWorker` вҶ’ `TelemetryUnpackerService` decodes, normalizes via `DeviceCalibration`, stores `TelemetryLog`.
8. **Streamr Broadcast** вҖ” `StreamrBroadcastWorker` publishes raw telemetry to Streamr P2P network (non-blocking, real-time forest pulse).
9. **peaq DID Verification** вҖ” `Peaq::DidRegistryService` verifies the tree's cryptographic machine passport (`did:peaq:0x...`).
10. **Server-Side Lorenz Verification** вҖ” `SilkenNet::Attractor.calculate_z` (BigDecimal, 18-digit precision) verifies device-computed Z-value (dual computation integrity).
11. **IoTeX ZK-Proof** вҖ” `IotexVerificationWorker` вҶ’ `Iotex::W3bstreamVerificationService` generates a zero-knowledge proof confirming data origin from real silicon.
12. **Chainlink Oracle Dispatch** вҖ” `ChainlinkDispatchWorker` вҶ’ `Chainlink::OracleDispatchService` bridges the ZK-verified proof to the blockchain via Chainlink Functions DON (guard clause: requires IoTeX verification first).
13. **Alert Dispatch** вҖ” `AlertDispatchService` checks 5 threat categories вҶ’ `EmergencyResponseService` dispatches actuators (water valves, fire sirens, seismic beacons).
14. **Tokenomics Evaluation** вҖ” `TokenomicsEvaluatorWorker` (hourly): 10,000 growth_points = 1 SCC вҶ’ `MintCarbonCoinWorker` вҶ’ `BlockchainMintingService` вҶ’ Polygon (Hadron KYC guard).
15. **Parallel Rewards** вҖ” `SolanaMicroRewardWorker` sends instant USDC micro-payments on Solana; `CeloRewardWorker` deposits cUSD community rewards on Celo.
16. **The Graph Indexing** вҖ” Every SCC mint event is indexed by The Graph subgraph, updating global carbon statistics via GraphQL.
17. **Daily Aggregation** вҖ” `DailyAggregationWorker` вҶ’ `InsightGeneratorService` вҶ’ `ClusterHealthCheckWorker` вҶ’ Slashing Protocol (if >20% trees anomalous) or `CeloRewardWorker` (if cluster healthy).
18. **KlimaDAO Retirement** вҖ” `KlimaRetirementWorker` retires SCC tokens via KlimaDAO for ESG carbon offset reporting.
19. **Filecoin Archive** вҖ” `FilecoinArchiveWorker` archives AuditLog to IPFS/Filecoin вҖ” immutable, decentralized, forever.
20. **Ethereum L1 Anchor** вҖ” `EthereumAnchorWorker` (weekly, Monday 03:00 UTC): SHA-256 state root (total_scc + chain_hash + timestamp) anchored to Ethereum Mainnet for ultimate finality.

## AI Oracle (Lorenz Attractor + ZK Verification)

The Lorenz attractor serves as a nonlinear transform of multi-sensor data into a single homeostasis indicator (Z-value). In Gaia 2.0, this is combined with IoTeX W3bstream ZK-proofs for trustless verification.

**Equations:**
```
dx/dt = Пғ(y - x)
dy/dt = x(ПҒ - z) - y
dz/dt = xy - ОІz
```

**Constants:** Пғ = 10.0, ПҒ = 28.0, ОІ = 8/3 (all BigDecimal, 18-digit precision)

**Perturbation:** Acoustic events shift Пғ (coupling, clamped 5вҖ“30), temperature shifts ПҒ (energy, clamped 10вҖ“50). The DID of each tree seeds unique initial conditions, creating a per-tree "chaotic fingerprint."

**Homeostasis check:** Z-value compared against per-species thresholds from `TreeFamily` (`critical_z_min` / `critical_z_max`). Trees within bounds earn growth points; trees outside bounds trigger drought/pest alerts.

**Dual Computation Integrity:** The same Lorenz equations run on-device (mruby, `firmware/bio_contracts/bio_contract.rb`) and on-server (`SilkenNet::Attractor`, BigDecimal). Divergence triggers fraud detection in `InsightGeneratorService`.

**Trustless Verification Pipeline:**
```
Lorenz Z-value (server) + hardware signature
    вҶ’ Iotex::W3bstreamVerificationService (ZK-proof generation)
    вҶ’ Chainlink::OracleDispatchService (decentralized bridge to blockchain)
    вҶ’ BlockchainMintingService (guard: verified_by_iotex? + oracle_status == "fulfilled")
```

## Security Model

- **Zero-Trust Architecture** вҖ” Every device has a unique AES-256 key stored in `HardwareKey` (encrypted at rest via ActiveRecord Encryption)
- **DID-Based Identity** вҖ” Each Soldier generates a DID from STM32 factory UID XOR'd with TRNG, locked in RTC backup registers. Registered as `did:peaq:0x...` via `Peaq::DidRegistryService` (self-sovereign machine passport)
- **ZK-Proof Verification** вҖ” IoTeX W3bstream validates that telemetry originated from real silicon hardware, not a software emulator
- **Chainlink Guard Clauses** вҖ” Decentralized oracle verification required before any token minting (prevents single-point-of-failure in the oracle layer)
- **Hadron KYC Compliance** вҖ” Polygon Hadron verifies investor KYC/KYB status before RWA token operations (ERC-3643)
- **Token Authentication** вҖ” Rails 8 `generates_token_for :api_access` for API, session-based for dashboard
- **RBAC** вҖ” Four roles: `investor` (read-only), `forester` (field operations), `admin` (full control), `super_admin` (system-level)
- **Encrypted Payloads** вҖ” AES-256-ECB for all LoRa and CoAP transmissions
- **Key Rotation** вҖ” `HardwareKeyService.rotate` generates new key + OTA downlink (Dual-Key Handshake with Grace Period)
- **Immutable Audit Trail** вҖ” SHA-256 chain_hash per organization вҶ’ Filecoin/IPFS archive (CID) вҶ’ weekly Ethereum L1 state root
- **BigDecimal Precision** вҖ” All financial and Lorenz calculations use 18-digit BigDecimal to ensure cross-platform determinism and legal-grade Web3 audit accuracy
